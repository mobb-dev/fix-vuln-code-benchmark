## What FINDING.txt described

An **OS Command Injection (CWE-78)** in yt-dlp's `--exec` data-flow, spanning `options.py` (option parsing) → `__init__.py` (postprocessor setup) → `YoutubeDL.py` (output-template substitution) → `postprocessor/exec.py` (the `Popen.run(cmd, shell=True)` sink), with `utils/_utils.py` (`shell_quote`) as the escaping primitive. (The line numbers/function labels in the finding were unreliable — the real signal was the file set and the vuln type.)

## Root cause

`ExecPP` runs its command with `shell=True`. Output-template fields are substituted into that command via `YoutubeDL.prepare_outtmpl`. The quoting-focused paths were all intact and secure:
- default (no field → `{}`) and `%(field)q` both go through `shell_quote` — I verified these do **not** inject.

But the **non-quoting conversions** — `%(title)s`, `%(field)j`, `%(field)l`, `%(field)r`, `%(field)a`, etc. — pass **untrusted remote metadata** (video title, uploader, tags…) straight into the shell **without escaping**. That is exactly the taint path the scanner flagged that *bypasses* `shell_quote`. A malicious title like `x; touch /tmp/PWNED` yields arbitrary command execution. I reproduced this end-to-end (a `PWNED` marker file was created).

## The fix

Added a `shell=True` mode to `prepare_outtmpl` that shell-escapes substituted values, and had `ExecPP.parse_cmd` request it. In `create_key`, when in shell mode, string-producing conversions (`s`, `j`, `l`, `r`, `a`, …) are wrapped with `shell_quote`. The `q` conversion is skipped (already quoted → no double-quoting), and numeric conversions are left as-is (digits can't inject).

## Why intended behavior is preserved

- The command still runs and still receives all the metadata as arguments — only the ability to *inject shell syntax* is removed.
- `q` and the default `{}`/filepath behavior are byte-for-byte unchanged (existing `TestExec.test_parse_cmd` still passes).
- The `shell` flag defaults to `False`, so **filename generation and `--print` are completely unaffected** (verified: `%(title)s` still produces raw output there, since those don't feed a shell).
